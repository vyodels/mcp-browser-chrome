# browser-mcp — 产品架构设计文档

## v3 隐蔽优先架构（当前版本，2026-04-04）

### 核心定位

项目目标：

- 不再做"扩展内置 AI Agent + 工作流平台"
- 做一套**隐蔽优先、MCP 优先、接近 Chrome MCP 的浏览器能力底座**
- 浏览器扩展只负责真实浏览器能力执行，不承载复杂业务编排
- 本地 MCP Server 负责把这些能力暴露给 Codex / LLM
- 具体业务流程（招聘、采集、表单处理）放在 Codex 侧完成

### 最高优先级项目规则

以下规则的优先级高于调试便利性、观测完整性和新功能接入速度：

1. **任何新增能力都不能破坏 stealth / anti-bot / anti-crawler 约束**
2. **默认执行路径不得扩大页面可探测面**
3. **凡是必须接触页面主 world 的能力，都只能显式触发、按需启用、最小化驻留**
4. **调试与观测必须服从隐蔽性目标，不能反过来塑造默认架构**
5. **默认禁止使用 Chrome debugger / CDP 调试通道**
6. **如果高风险实现被收紧，必须同步给出公开方法的替代路径或安全降级行为，不能直接让链路断掉**

整体链路：

```
Codex / LLM
    ↓ MCP stdio
Local MCP Server（mcp/server.mjs）
    ↓ Unix socket
Native Messaging Host（native-host/host.mjs）
    ↓ Chrome Native Messaging
Chrome Extension Background（src/extension/background/）
    ↓
Content Script（src/extension/content/）
    ↓
真实网页（共享用户登录态）
```

---

### 为什么选 Native Messaging

本项目首要目标之一是降低被网页侧反爬虫 / 反自动化机制探测到的概率。扩展与本地 MCP Server 的通信优先选择 Chrome Extension Native Messaging，而不是：

- 本地 HTTP Server（暴露 `localhost` 端口）
- 本地 WebSocket Server（暴露 `ws://` 连接）
- 页面内桥接脚本 + `window.postMessage`

Native Messaging 的优势：

1. 页面 JS 无法探测本地开放端口
2. 不暴露 `localhost` / `127.0.0.1` / `ws://` 等典型辅助进程特征
3. 通信链路完全在浏览器扩展与本地宿主之间，无页面可见的网络痕迹
4. 更适合长期产品化与稳定运行

注意：Native Messaging 能显著降低"桥接层被探测"的风险，但不能保证绝对不可发现。更容易暴露的往往是页面内的事件链异常和环境污染，而不是本地桥接。

---

### 禁止的实现方式

以下实现方式与"隐蔽优先"目标冲突，默认禁止：

1. **不允许使用本地 HTTP / WebSocket 服务作为扩展桥接主通道**
2. **不允许在页面主世界长期注入桥接脚本**（不允许挂载 `window.__MCP__` 等对象，不允许通过 `CustomEvent` / `postMessage` 暴露控制协议）
3. **不允许污染页面全局环境**（不修改原型链，不持久改写 `fetch` / `XHR` / `console`，不插入可见 overlay）
4. **不允许默认高频全量扫描 DOM**（页面读取分层：snapshot → query → debug）
5. **不允许把业务流程固化在扩展内**（扩展只保留浏览器能力，不承载业务逻辑）
6. **不允许在 content script 默认启动路径里注入页面主 world 诊断脚本**（例如默认 `console` monkey-patch；这类能力只能由显式诊断请求按需开启）
7. **不允许在真实站点上通过 `chrome.debugger`、CDP attach、debug 模式仿真等方式实现默认能力**

---

### 内容脚本低痕执行原则

1. **优先使用 content script isolated world**，主要逻辑不向页面主 world 注入
2. **仅在需要时做最小交互**：先读页面，再局部定位，再执行操作，失败时才进入 debug 模式
3. **事件链尽量接近真人**：
   - click 补齐 `pointerover → mouseover → mousemove → pointerdown → mousedown → focus → pointerup → mouseup → click`
   - fill 使用原生 setter，触发 `input` / `change` / `blur`
4. **操作节奏反规律化**：随机延迟存在但不机械，滚动距离、输入间隔、点击前停顿均需随机化
5. **默认只暴露必要快照**：页面文本限制长度，交互元素限制数量，HTML 仅在 debug 模式下返回
6. **显式等待优于盲等**：优先 `wait_for_element` / `wait_for_text` / `wait_for_navigation`
7. **观测能力默认后置**：像页面 `console` 采集这类能力，必须由显式请求触发，且优先选择被动监听而不是 monkey-patch
8. **公开方法必须有安全降级**：例如点击/输入要优先走语义操作和原生 setter，不能因为禁用高风险实现而直接变成不可用
9. **页面交互前必须先做层级清障**：遇到弹窗、下载引导、浮层、全屏蒙层、置顶提示条等遮挡主内容的元素时，应先识别并关闭，再继续业务操作；不能忽略遮挡直接点击底层元素
10. **不得抢占用户正在看的前台标签页**：真实站点默认不应依赖“当前活动标签页”兜底，也不应为了读取、截图或定位而偷偷切换用户当前正在看的标签页
11. **默认要求显式 `tabId`**：除标签管理类工具外，AI 在真实页面执行操作前应先枚举/确认目标标签页，再把 `tabId` 作为显式输入传下去
12. **定位优先收缩作用域**：列表、卡片、聊天记录、iframe 内重复结构应先缩小上下文，再按 `ref/selector/role/text` 取目标，不应直接依赖全页第一个文本命中
13. **大文本读取必须分层分段**：先消费结构化摘要和分页信息，需要更多内容时再继续抓取下一段，而不是默认抓完整 HTML 或整页全文
14. **真实站点点击默认保守**：不要在一次用户意图下自动升级成 root / ancestor 多次补点；真实页面点击应优先“单次精确点击 + 显式等待业务信号”
15. **一键副作用动作必须显式放行**：`打招呼 / 继续沟通 / 发送 / 求简历 / 换电话 / 换微信 / 索要联系方式` 这类动作默认不应直接执行，必须由上层 AI 明确 opt-in

### MCP 工具设计原则

1. **工具分层，不走"大一统万能工具"**：tabs/navigation、snapshot/query、interaction、wait、capture/files
2. **默认最小读取**：`browser_snapshot` 返回精简信息，`browser_debug_dom` 仅按需调用
3. **交互工具支持多种定位方式**：`ref`、`selector`、`text`、`role`、`index`，优先级从高到低
4. **工具返回结构化结果**：至少包含 `success`、`tabId`、`target`、`navigationDetected`、`snapshotSummary`、`error`
5. **默认不截图**：`browser_take_screenshot` 为显式工具，不作为常规读取手段
6. **默认不抓完整 HTML**：全量 DOM 视为调试级工具，与正常 snapshot 分离
7. **工具使用顺序要符合真人操作**：推荐采用 `snapshot/query -> 判断是否有 overlay/dialog/toast/遮挡 -> 先关闭或取消 -> 再进行业务点击/输入` 的顺序
8. **高风险交互必须有备选方案**：对于原生 dialog、整页截图、debug-only 观测、复杂 hover/click 等高风险路径，工具设计必须同步给出结构化替代方案或人工接管点，不能只给“禁用”
9. **工具返回应便于 AI 继续编排**：当某条路径因 stealth 约束不能默认执行时，返回值中应优先给出替代建议、降级模式或下一步推荐，而不是只返回失败
10. **高副作用点击必须带风险闸门**：真实站点上的一键业务动作应在工具层返回结构化 risk 信息，并要求显式放行，而不是和普通跳转按钮共用默认 click 语义

### AI 使用约束

给 AI 调用方的默认执行顺序如下：

1. 先通过 tabs 类工具确认目标页面，并显式拿到 `tabId`
2. 先做轻量读取：优先 `snapshot`，必要时再 `query`
3. 判断是否存在遮挡、置顶弹层、下载提示、toast、引导层
4. 如果有遮挡，先关闭、取消或收起，再继续底层业务交互
5. 选择 locator 时优先 `ref`，其次稳定 `selector`，再次是 `role + text`，最后才是纯文本模糊匹配
6. 读取长文本、长列表、聊天记录时优先消费分页/摘要元数据，再按段继续读取
7. 遇到高风险交互时优先走替代方案或安全降级，不直接宣告链路不可用
8. 遇到 `打招呼 / 发送 / 求简历 / 换电话 / 换微信 / 索要联系方式` 这类动作时，先判断是否属于真实业务提交；若是，默认停在风险提示，等待显式放行

高风险交互的典型备选方向：

1. 原生 JS dialog：优先页面内自定义弹层，仍不安全时转人工接管
2. 整页截图：优先元素截图或当前可见区域截图，整页流程优先在 mock 页面验证
3. debug-only 观测：优先被动读取和结构化元数据，不进入 Chrome debugger / CDP
4. 复杂点击失败：先缩小作用域、先清障、先滚动到可见，再重试更接近真人的交互链

### 能力边界

#### 扩展内保留的能力

- 标签页识别与切换
- 页面导航（打开、后退、刷新）
- 页面快照与元素查询
- 点击 / 悬停 / 输入 / 清空 / 选择 / 滚动 / 按键
- 显式等待（元素出现 / 文本出现 / 导航完成 / 元素消失）
- 截图
- 文件下载与保存
- 基础风控节奏控制（`src/rateLimit.ts`）

#### 已从扩展中移出的能力（放到 Codex / MCP client 侧）

- 扩展内置 LLM 对话主循环
- 工作流引擎
- 业务场景模板（招聘、小红书等）
- 子代理系统
- 持久记忆与会话记忆系统
- Skill 自动沉淀与审批流
- 候选人 CRM / 工作区 schema 系统

---

### 验收标准

以下能力稳定跑通，才算进入可用状态：

1. Codex 能枚举并选择浏览器标签页
2. 能读取页面快照而不依赖截图
3. 能通过 `ref` / `selector` / `text` 三种方式定位并操作元素
4. 能通过显式 wait 工具稳定等待页面变化
5. 能在真实登录态页面上执行多步骤流程
6. 能把结果保存到本地文件
7. 整个链路不依赖 localhost HTTP / WebSocket

---

### 最小验证流程

先用小场景验收，不要从完整业务流程切入：

1. 打开指定页面
2. 定位搜索框
3. 输入关键词
4. 等待结果列表出现
5. 提取前 10 条标题和链接
6. 保存为 JSON 文件

此流程稳定通过后，再进入招聘、沟通、简历处理等更复杂场景。
