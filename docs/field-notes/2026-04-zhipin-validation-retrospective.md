# zhipin.com 真实验证复盘

日期：`2026-04-21`

## 目的

本轮验证的目标不是大范围自动化，而是确认当前 `browser-mcp` 在真实 `zhipin.com` 上，是否能够以较低风险完成以下主链路：

- 通过侧边栏和页面内入口进入职位管理、推荐牛人、沟通等页面
- 读取职位列表、推荐候选人列表、在线简历、聊天内容等页面级数据
- 验证页面层级、frame 结构、遮挡层处理与基础交互能力

## 已确认的事实

### 1. 页面主流程不是 canvas-first

在本轮抓到的页面中，主业务链路主要是普通 DOM 与 iframe：

- 职位管理壳页与职位详情页
- 推荐牛人壳页
- 推荐牛人的业务 frame：`/web/frame/recommend/...`
- 在线简历侧栏
- 风控后的安全提示登录页

这些页面都能直接读取：

- 文本
- `button / a / input / li / div / span`
- frame 结构
- 候选人姓名、`打招呼`、侧边栏 `沟通` 等可交互元素

结论：主流程不是依赖整页 `<canvas>` 承载内容。

### 2. 推荐牛人页面采用“壳页 + 业务 frame”结构

已确认当前推荐页结构中至少存在：

- `@main`：壳页，包含左侧边栏与外层导航
- `@frame:523`：业务 frame，承载推荐列表与在线简历侧栏

结论：后续任何推荐页验证都应先识别 frame，再决定在壳页还是业务 frame 内操作。

### 3. 候选人姓名可展开在线简历

在推荐页业务 frame 中，直接点候选人姓名文本，能够使当前候选人的在线简历侧栏展开。

这说明：

- “整个候选人卡片可点击”不是可靠前提
- 在线简历入口更可能是卡片内某个更细粒度的目标，而不是外层 `li`

### 4. `打招呼` 是真实业务副作用动作

在在线简历侧栏中点击主 `打招呼` 按钮后，页面状态发生了真实变化：

- 当前主按钮从 `打招呼` 变成 `继续沟通`
- 页面中出现“为你推荐 与王相杰相似的15个牛人”之类后态文本

结论：这里不是纯页面跳转，而是会直接发起站内业务状态变更的一键动作。

### 5. 触发风控后，目标页被重定向到了安全提示登录页

后续尝试通过侧边栏进入 `沟通` 时，页面落到了：

- `https://www.zhipin.com/web/user/?ka=header-login`

页面文案明确说明：

- 账号被检测到存在使用第三方招聘管理系统、插件、外挂、软件等辅助工具
- web 端登录被暂时限制

这是本轮验证中最明确、最关键的站点反馈。

## 触发风控前的关键操作链

按时间顺序，关键动作大致是：

1. 在推荐牛人壳页识别业务 frame
2. 在业务 frame 中定位候选人
3. 点击候选人姓名，展开在线简历
4. 在在线简历区域继续尝试 `打招呼`
5. 随后通过侧边栏尝试进入 `沟通`
6. 页面被安全策略拦截并跳到登录限制页

这里有两个关键观察：

- 前几步都只涉及局部状态变化，比较容易“看起来成功”
- 真正高风险的是“高副作用业务动作 + 随后继续推动站内状态切换”

## 原因分析

以下是本轮复盘后最可信的原因排序。

### 1. 点击执行模型仍然是 synthetic DOM 事件

当前工具的点击主链路仍然基于内容脚本里的事件分发与 `.click()` 语义触发，而不是真正的浏览器可信输入。

这意味着：

- 页面看到的事件链不够像真实用户
- `event.isTrusted` 不是当前路线的强项
- 即使延迟随机化，也掩盖不了“事件来自脚本”的核心事实

### 2. 点击 fallback 过于激进

本轮实测前，点击链路存在这样的风险：

- 目标点不生效时，继续尝试 `activationTarget`
- 再继续尝试 `root`
- 甚至尝试祖先节点

在异步路由、局部刷新、frame 页面和侧边栏切换场景里，这非常容易把“一次用户意图”放大成多次激活。

### 3. 高副作用按钮没有单独风控

`打招呼`、`继续沟通`、`求简历`、`换电话`、`换微信` 这类动作，在真实业务上不应和普通页面导航按钮共用默认 click 语义。

本轮在这里吃到的不是“识别不准”，而是“动作级别判断不够保守”。

### 4. 文本定位在重复结构页面里仍然有误击风险

推荐页这类页面具有：

- 重复卡片
- 重复按钮文案
- 外层壳页与内层 frame 共存

如果不先缩小作用域，就很容易把：

- 候选人姓名
- 卡片容器
- 壳页侧边栏
- 业务 frame 内按钮

混成一组同名命中结果。

## 本轮已经落地的修正

以下修正已经进入代码并完成构建：

### 1. 真实站点默认改为保守点击

`browser_click` 在真实站点默认不再自动升级到：

- `root`
- `ancestor`

这意味着点击失败时默认只保留“单次精确点击”，而不是自动补点。

### 2. 高副作用动作默认拒绝执行

以下文案被纳入高风险动作识别：

- `打招呼`
- `继续沟通`
- `发送`
- `求简历`
- `换电话`
- `换微信`
- `索要联系方式`

默认会返回风险提示，只有显式传入 `allowDangerousAction=true` 才允许执行。

### 3. 点击后的观察窗口拉长

默认观察窗口从较短时间提升到了更保守的水平，目的是减少：

- 页面异步状态尚未更新
- 工具误判为“没反应”
- 紧接着又触发补救点击

### 4. 工具说明和项目规范已同步更新

这类约束不再只存在于口头策略里，而是已经下沉到：

- MCP tool 描述
- `README.md`
- `docs/project-prompt.md`

## 后续回归必须遵守的规则

在 `zhipin.com` 限制窗口结束后，如需继续真实验证，应遵守以下顺序。

### 1. 先只读，后交互

先做：

- `browser_tabs`
- `browser_frames`
- `browser_snapshot`
- `browser_query_elements`

不要一上来就点页面。

### 2. 先判层级，再判作用域

在壳页 + frame 场景里，必须先确认：

- 目标在壳页还是业务 frame
- 当前是否有浮层、下载提示、遮挡层、弹窗

### 3. 优先新鲜 `ref`，其次局部文本

在重复列表页中：

- 优先刚拿到的新鲜 `ref`
- 其次使用已缩小作用域的 `role + text`
- 最后才考虑全页纯文本定位

### 4. 不默认执行一键副作用动作

以下动作默认停在风险提示：

- `打招呼`
- `继续沟通`
- `求简历`
- `换电话`
- `换微信`

只有在明确需要验证真实提交链路时，才允许显式放行。

### 5. 不再用真实站点验证“高副作用动作是否可点”

这类动作的默认回归应转移到：

- mock 页面
- 离线结构页
- 纯静态还原页

真实站点只验证：

- 页面读取
- 页面结构识别
- 低风险导航入口
- 会话内容的只读抓取

## 未来高价值升级项

### 1. 在快照里补“遮挡摘要”

快照结果中建议补充：

- top layer 候选
- 可见遮挡区域
- z-index 高的固定层
- 候选关闭按钮

### 2. 在点击失败时补“怀疑的阻挡者”

建议在点击失败返回里补：

- 点击点附近的 `elementsFromPoint` 栈
- suspected blocker
- 目标与阻挡者是否同 frame

### 3. 对重复结构页面强化“局部作用域定位”

如推荐页、沟通页、聊天列表、候选人列表：

- 先卡片
- 再卡片内目标

不要让全页文本命中成为默认方案。

### 4. 真实站点进一步收缩到“低探测只读模式”

如果站点风险仍高，真实环境默认能力应进一步限定为：

- frame 识别
- 结构快照
- 文本快照
- 低频局部查询

## 结论

本轮复盘的核心结论不是“随机延迟还不够”，而是：

- 当前 DOM synthetic 输入模型不适合作为真实强风控站点的默认交互模型
- 页面验证和业务动作验证必须分开
- 高副作用动作必须在工具层单独建闸
- 对 `zhipin.com` 这类站点，主策略应从“尽量点通”切换为“先只读、后人工、mock 承担副作用链路”
